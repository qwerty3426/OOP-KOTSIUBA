# Лабораторна робота №17

## Тема: Породжувальні патерни: Factory і Singleton

### 1. Вступ
Патерни проєктування (Design Patterns) — це типові рішення для часто виникаючих проблем у проектуванні ПЗ. Вони дозволяють повторно використовувати перевірені рішення, покращують комунікацію між розробниками та допомагають створювати гнучкі й підтримувані системи.

Патерни поділяються на:
- **Породжувальні (Creational)**: стосуються створення об’єктів.
- **Структурні (Structural)**: стосуються композиції класів та об’єктів.
- **Поведінкові (Behavioral)**: стосуються взаємодії та розподілу відповідальностей між об’єктами.

---

### 2. Патерн Factory Method (Фабричний метод)
**Призначення:** Визначає інтерфейс для створення об’єкта, але дозволяє підкласам вирішувати, який клас інстанціювати.  

**Проблема:** Коли клас не може передбачити, які об’єкти йому потрібні, або хоче делегувати створення об’єктів іншим класам.  

**Структура:**
- `Product` – інтерфейс або абстрактний клас для продуктів.
- `ConcreteProduct` – конкретна реалізація продукту.
- `Creator` – оголошує фабричний метод.
- `ConcreteCreator` – перевизначає метод для створення конкретного продукту.

**Переваги:**
- Додавання нових продуктів не змінює клієнтський код (OCP).  
- Слабке зв’язування з конкретними класами.

---

### 3. Патерн Singleton (Одинак)
**Призначення:** Гарантує наявність тільки одного екземпляра класу та глобальну точку доступу до нього.  

**Проблема:** Коли потрібен лише один екземпляр ресурсу (напр., менеджер конфігурації).  

**Структура:**
- Приватний статичний екземпляр.
- Приватний конструктор.
- Публічний статичний метод або властивість для доступу до екземпляра.

**Переваги:**
- Контрольований доступ до єдиного екземпляра.  
- Економія ресурсів.  
- Глобальна точка доступу.  

**Недоліки:**
- Може ускладнити тестування.  
- Порушує принцип єдиної відповідальності (SRP).  
- Жорстка зв’язаність з класом.
